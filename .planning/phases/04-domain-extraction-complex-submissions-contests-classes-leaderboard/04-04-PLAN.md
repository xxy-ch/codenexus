# Plan 04: Create domain-leaderboard Crate + SEC-03 Tenant Fix

<objective>
Extract the leaderboard module from the api crate into an independent `domain-leaderboard` workspace crate AND fix the SEC-03 tenant isolation vulnerability. This is the most complex extraction because:

1. **Cross-domain dependency**: leaderboard routes import `crate::classes::service::ClassService` -- resolved via the `ClassMembershipChecker` trait created in Plan 01 (Wave 1).
2. **SEC-03 fix**: `get_global_leaderboard` and `get_problem_leaderboard` currently expose cross-tenant data. The `_claims` parameter is discarded. Fix: extract claims, pass org filter to service, modify SQL, scope cache keys.
3. **LeaderboardService normalization**: Currently creates its own `redis::Client` per call. Normalize to accept `Option<deadpool_redis::Pool>` from AppState.

Purpose: Complete the final and most complex domain extraction. Fix a security vulnerability that allows any authenticated user to see cross-tenant leaderboard data.

Output: Independent `domain-leaderboard` crate with SEC-03 fix applied.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Client -> Leaderboard API | Untrusted input (query params, path params). Tenant ID from JWT claims (trusted). |
| Leaderboard API -> Database | SQL with tenant filter. Parameterized queries prevent injection. |
| Leaderboard API -> Redis Cache | Cache keys scoped by org_id prevent cross-tenant leakage. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-05 | I (Information Disclosure) | get_global_leaderboard SQL | mitigate | SEC-03: Add `WHERE u.organization_id = $N` when school_id is Some. Non-admin users only see their org. Root/OrgAdmin bypass via school_id=None. |
| T-04-06 | I (Information Disclosure) | get_problem_leaderboard SQL | mitigate | SEC-03: Add `AND ($2::BIGINT IS NULL OR u.organization_id = $2)` to existing WHERE clause. Same role bypass logic. |
| T-04-07 | T (Tampering) | Leaderboard Redis cache keys | mitigate | Scope cache keys with org_id: `leaderboard:global:{org_id}:{limit}:{offset}`. Prevents org A's cached data being served to org B. |
| T-04-08 | E (Elevation of Privilege) | is_admin role check | accept | Role parsed from JWT claims (cryptographically signed). Cannot be forged by client. |
| T-04-09 | I (Information Disclosure) | LeaderboardService Redis client | mitigate | Normalize from `redis::Client` to `deadpool_redis::Pool` from AppState. Prevents unbounded connection creation (SEC-05 alignment). |
</threat_model>
