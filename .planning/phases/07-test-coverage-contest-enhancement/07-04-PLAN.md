    **Create `api/tests/tenant_isolation.rs`:**
    This is the critical security test suite (TEST-03). Pattern:
    1. Spin up TestFixture with real PostgreSQL + Redis
    2. Seed data in TWO organizations (org_id=1, org_id=2)
    3. For each endpoint group, make the same query scoped to org 1 and verify org 2 data never appears

    Tests:
    1. `test_contest_list_tenant_isolated` -- create contests in both orgs, query org 1, verify only org 1 contests returned
    2. `test_problem_list_tenant_isolated` -- create problems in both orgs, query org 1, verify only org 1 problems returned
    3. `test_user_list_tenant_isolated` -- create users in both orgs, query org 1, verify only org 1 users returned
    4. `test_submission_list_tenant_isolated` -- create submissions in both orgs, query org 1, verify only org 1 submissions
    5. `test_leaderboard_global_tenant_isolated` -- seed users + submissions in both orgs, query leaderboard for org 1, verify only org 1 users in rankings

    **Important notes:**
    - These are integration-level tests that call the domain service layer directly (not HTTP handlers), because tenant filtering happens in the SQL queries within service functions. Handler-level tenant tests would duplicate the auth middleware tests.
    - Use `TestFixture::new()` + `run_migrations()` from api-infra testkit.
    - Seed data with explicit `organization_id` values to control tenant membership.
    - For each test, seed identical data in both orgs, then verify the query result only contains data from the queried org.
