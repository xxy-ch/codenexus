<task id="1">
<title>Create Gateway HTTP client with TTL cache</title>
<action>
Create or add to a new module (e.g., `backend/api-infra/src/feature_gateway/client.rs`):

1. Define `GatewayClient` struct:
   ```rust
   pub struct GatewayClient {
       http: reqwest::Client,
       base_url: String,
       cache: Arc<DashMap<String, CachedResolve>>,
       auth_header: String,
   }
   struct CachedResolve {
       resolved: ResolvedFeature,
       expires_at: Instant,
   }
   ```

2. `GatewayClient::new(base_url: String, worker_secret: String)` — creates reqwest::Client, stores Bearer token

3. `GatewayClient::resolve(&self, slug: &str, campus_id: Option<i64>, grade_id: Option<i64>) -> ResolvedFeature`:
   - Build cache key: `"{slug}:{campus_id:?}:{grade_id:?}"`
   - Check cache: if entry exists and `Instant::now() < expires_at` → return cached
   - HTTP call: `GET {base_url}/resolve?slug={slug}&campus_id={cid}&grade_id={gid}` with `Authorization: Bearer {secret}`
   - On success: cache result with 10s TTL (`Instant::now() + Duration::from_secs(10)`)
   - On failure (connection refused, timeout 2s): log `warn!("feature gateway unavailable, fail-open for slug={}", slug)`, return `ResolvedFeature { enabled: true, source: Default }` (D-23 fail-open)

4. Keep `ResolvedFeature` and `FeatureSource` types available (they can be duplicated locally or referenced from the feature-gateway crate via shared dependency)

5. Keep using `reqwest` (already in api Cargo.toml dependencies)
</action>
<acceptance_criteria>
- `GatewayClient` struct exists with `http`, `base_url`, `cache`, `auth_header` fields
- `resolve()` checks TTL cache before HTTP call
- `resolve()` caches successful results with 10s TTL
- `resolve()` returns enabled=true on HTTP failure (fail-open D-23)
- `resolve()` sends `Authorization: Bearer` header (D-24)
- Unit test: cache hit returns cached value without HTTP call
- Unit test: HTTP failure returns enabled=true
</acceptance_criteria>
</task>

<task id="3">
<title>Convert API feature routes to Gateway proxies</title>
<action>
Replace `backend/api-infra/src/feature_gateway/routes.rs` route handlers:

1. `resolved_features` handler: call `GET {gateway_url}/resolve?...` with campus_id/grade_id from TenantContext, return response
2. `list_registry` handler: proxy `GET {gateway_url}/registry`
3. `list_flags` handler: proxy `GET {gateway_url}/features/{slug}/flags`
4. `set_flag` handler: proxy `POST {gateway_url}/features/{slug}/flags` with same body
5. `delete_flag` handler: proxy `DELETE {gateway_url}/features/{slug}/flags?scope=...&scope_id=...`

All proxies:
- Forward the caller's `Authorization: Bearer {token}` from the original request (for role checks on Gateway)
- Alternatively, use the internal WORKER_SECRET for API-to-Gateway calls (simpler)
- Return Gateway's JSON response as-is
- On Gateway failure: return 502 Bad Gateway with error message

Keep `check_scope_authorization` as a local function if still needed for early validation, or remove it if Gateway handles all auth.
</action>
<acceptance_criteria>
- `features_router()` no longer references `FeatureGatewayService` or `AppState.feature_gateway`
- Each handler proxies to Gateway via HTTP using GatewayClient
- Response format unchanged from frontend perspective

</acceptance_criteria>
