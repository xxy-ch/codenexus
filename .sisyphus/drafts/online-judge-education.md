## 研究结果

### OnlineJudge系统架构（已完成）
**推荐架构**：三层架构 + 微服务模式
- **前端**：React/Vue + TypeScript
- **后端API**：Django/Go（用户选择Rust + Axum）
- **评测Worker**：独立的服务，负责代码执行
- **数据库**：PostgreSQL + Redis缓存
- **消息队列**：BullMQ（Redis）或RabbitMQ
- **隔离方式**：用户选择原生进程 + chroot/cgroups

**核心组件**：
1. 题目管理服务（CRUD题目、测试用例）
2. 提交服务（接收代码、验证、排队）
3. 评测服务（Worker，执行代码沙箱）
4. 用户服务（认证、角色管理）
5. 竞赛/作业服务（竞赛调度、排名）
6. 通知服务（WebSocket实时更新）

**流行开源实现**：
- DOMjudge（PHP + MySQL，ICPC竞赛）
- DMOJ（Django + PostgreSQL）
- QingdaoU（Django + MySQL + Vue）
- Judge0（Go + Docker）
- Hydro（Python + MongoDB）

### 教育系统权限设计（已完成）
**推荐模式**：Tenant ID列 + 应用层过滤
- **数据库模式**：所有表包含organization_id
- **中间件**：自动设置当前租户上下文
- **自动过滤**：TenantAwareModel自动过滤所有查询

**角色定义**：
```rust
enum Role {
    Root,              // 平台管理员
    CampusAdmin,       // 校区管理员
    Teacher,           // 教师
    Student,           // 学生
}
```

**权限委托**：
- 层级委托：Root → Campus → Teacher → Student
- 作用域检查：校区管理员只能管理自己校区
- 教师学生关系：ClassEnrollment中间表

**数据隔离策略**：
1. Tenant ID列（推荐用于小型系统）
2. PostgreSQL Row-Level Security（企业级）
3. Separate Schemas（超大型系统）

### 评测系统实现（已完成）

**核心架构**：
- **评测流程**：提交 → 队列 → Worker编译 → 沙箱执行 → 结果存储 → 通知
- **队列管理**：Celery + Redis（或Rust原生队列）
- **并发控制**：多Worker，CPU核心绑定，公平调度

**沙箱隔离**（用户选择原生进程 + chroot/cgroups）：
- **Linux Cgroups v2**：精确资源限制（CPU、内存）
- **Chroot环境**：文件系统隔离
- **Seccomp-BPF**：系统调用过滤
- **避免**：Docker特权模式（有CVE漏洞）

**评测结果类型**（按优先级排序）：
| 代码 | 优先级 | 说明 |
|------|--------|------|
| IE | 最高 | 系统内部错误 |
| RTE | 高 | 运行时错误（段错误、除零等） |
| OLE | 高 | 输出超限（>256MB） |
| MLE | 高 | 内存超限 |
| TLE | 高 | 时间超限 |
| WA | 中 | 答案错误 |
| AC | 低 | 全部通过 |

**测试用例管理**：
- Kattis Problem Package格式
- Sample（示例）+ Secret（隐藏）测试
- 懒惰求值：遇到高优先级错误立即停止
- Output Validator支持浮点数容差、多解等

**防作弊实现**：
- 代码相似度检测：MOSS算法或AST比对
- 提交时间分析：检测异常同时提交
- 运行特征分析：时间/内存使用一致性
- 教师人工审核：最终判定权在教师

**性能优化**：
- 懒惰评测：失败即停止，节省资源
- 结果缓存：Redis缓存最近提交
- 编译缓存：重复提交跳过编译步骤

**安全检查清单**：
- ❌ 绝不使用Docker特权模式
- ✅ 使用cgroups v2准确资源限制
- ✅ seccomp-BPF系统调用过滤
- ✅ 网络隔离（禁止出站）
- ✅ 文件系统只读
- ✅ 定期安全审计
