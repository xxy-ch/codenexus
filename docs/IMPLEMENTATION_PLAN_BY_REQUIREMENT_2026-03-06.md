## 0. 计划原则

- 先打通主链路，再扩展功能。
- 每个阶段必须有验收标准（DoD）。
- 以真实 API 为准，逐步清理 mock fallback。

## 1. 需求拆解（默认优先级）

P0（必须）：
- 认证可用（登录/注册/鉴权）
- 判题链路可用（提交 -> worker -> 回写）
- 前后端接口对齐（至少覆盖主流程）

P1（应做）：
- 搜索、讨论、博客、排行榜主页面可用
- 通知系统基础可用（列表 + 已读）

P2（可延后）：
- 内容版本化/审核
- 声誉与徽章
- 导出能力

## 2. 分阶段计划

## Phase A：一致性修复（1-2 天）

任务：
- 校正路由鉴权作用域，确保公开接口可访问。
- 对齐前端 service 路径与后端实际路由（先覆盖核心模块）。
- 清理“API 失败自动回退 mock”的高风险分支（保留可控开关，不做静默回退）。

DoD：
- 手工调用登录/注册接口返回预期状态码。
- 前端核心页面无 404/401 误报（由路径不一致导致）。

## Phase B：主链路验收（2-3 天）

任务：
- 认证 -> 题目 -> 提交 -> 判题 -> 结果展示全链路联调。
- 为提交与判题回写补充最小集成测试。
- 增加最小 E2E 冒烟（1 条主流程用例）。

DoD：
- 冒烟测试稳定通过。
- 至少 1 条真实提交从 pending 到最终状态可见。

## Phase C：需求化扩展（3-5 天）

任务：
- 按需求优先级逐项恢复模块（搜索/社区/排行榜/通知）。
- 对每个模块补最小单测或集成测试。
- 新增变更日志，记录接口改动与前端适配点。

DoD：
- 每个启用模块都具有“可访问接口 + 可用前端入口 + 最小测试”。

## 3. 需求输入模板（后续每次迭代都用）

- 业务目标：本次要支持的用户动作是什么？
- 角色范围：普通用户 / 教师 / 管理员？
- 验收方式：API 验证、页面操作还是自动化测试？
- 截止时间：硬截止日期。

## 4. 本周建议执行顺序

1. Phase A：先修一致性（鉴权 + 路径 + mock 回退）。
2. Phase B：打通一条可演示主链路。
3. Phase C：再进入功能扩展。
